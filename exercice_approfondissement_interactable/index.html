    Exercice d'approfondissement - Suppression via XR Interactable et Interaction Layers

    Objectifs

    Dans l'exercice principal, la suppression des cubes peints utilise un Physics.Raycast depuis le contrôleur gauche. Dans cet exercice, vous allez remplacer cette approche par les outils du toolkit — en exploitant les événements du XR Simple Interactable et les Interaction Layers pour un résultat plus propre et plus robuste.

    Concrètement, vous allez :

    1. ✅ Ajouter XR Simple Interactable sur le prefab CubePeinture
    2. ✅ Configurer des Interaction Layers pour séparer les cubes peints des cubes de couleur
    3. ✅ Brancher la suppression via selectEntered dans l'Inspector — sans écrire de script
    4. ✅ Simplifier PeintureVR en retirant le raycast de suppression devenu inutile

    Prérequis : Avoir complété l'exercice de peinture VR principal

    Pourquoi c'est mieux

    Dans l'exercice principal, supprimer un cube nécessite un Physics.Raycast manuel qui vérifie le tag CubePeint. Ça fonctionne, mais c'est redondant : le Near-Far Interactor sait déjà quel objet est pointé. Autant laisser le toolkit gérer cette logique.

    Aspect Exercice principal Cet exercice
    Détection du cube ciblé Physics.Raycast + tag Near-Far Interactor
    Déclenchement action.performed gâchette gauche selectEntered toolkit
    Code nécessaire Méthode OnGachetteGauche Aucun script
    Feedback visuel Aucun — on ne sait pas si on pointe le cube Ligne du Near-Far Interactor qui s'arrête sur le cube

    Étape 1 : Créer les Interaction Layers

    Les Interaction Layers permettent de définir quels interactors peuvent interagir avec quels interactables. Sans ça, le Near-Far Interactor répondrait à tous les XR Interactable de la scène — cubes de couleur et cubes peints confondus — ce qui causerait des conflits avec la logique de peinture.

    1.1 Ajouter les layers

    1. Edit → Project Settings → XR Interaction Toolkit
    2. Vous voyez la liste des Interaction Layers — Default est déjà présent

    3. Ajoutez deux nouveaux layers :

    4. Couleur — pour les cubes de sélection de couleur

    5. Peint — pour les cubes placés sur le mur

    1.2 Assigner le layer aux cubes de couleur

    Pour chacun des 4 cubes de couleur flottants dans la scène :

    1. Sélectionnez le cube
    2. Sur le composant XR Simple Interactable, trouvez Interaction Layer Mask
    3. Décochez Default et cochez Couleur

    1.3 Configurer le Near-Far Interactor du contrôleur droit

    Le contrôleur droit gère la peinture et la sélection de couleur — il doit répondre uniquement au layer Couleur.

    1. Dans la Hierarchy, dépliez XR Origin (XR Rig) → Camera Offset → Right Controller → Near-Far Interactor
    2. Sur le composant XR Near-Far Interactor, trouvez Interaction Layer Mask
    3. Décochez Default et cochez Couleur

    1.4 Configurer le Near-Far Interactor du contrôleur gauche

    Le contrôleur gauche gère la suppression — il doit répondre uniquement au layer Peint.

    1. Dépliez XR Origin (XR Rig) → Camera Offset → Left Controller → Near-Far Interactor
    2. Sur Interaction Layer Mask, décochez Default et cochez Peint

    Ce qu'on obtient : Le contrôleur droit ne voit que les cubes de couleur. Le contrôleur gauche ne voit que les cubes peints. Les deux types d'objets sont complètement isolés l'un de l'autre du point de vue du toolkit.

    Étape 2 : Modifier le prefab CubePeinture

    2.1 Ajouter XR Simple Interactable

    1. Dans le dossier Assets/Prefabs, double-cliquez sur CubePeinture pour l'ouvrir en mode édition de prefab
    2. Add Component → XR → XR Simple Interactable
    3. Sur le composant, trouvez Interaction Layer Mask
    4. Décochez Default et cochez Peint

    2.2 Brancher la suppression dans l'Inspector

    Toujours sur le composant XR Simple Interactable :

    1. Trouvez la section Interactable Events → dépliez Select → trouvez Select Entered
    2. Cliquez sur le + pour ajouter une entrée
    3. Glissez le GameObject racine du prefab (le cube lui-même) dans le champ objet
    4. Dans le menu déroulant de la fonction, sélectionnez GameObject → SetActive

    Attendez — SetActive(false) n'est pas une suppression. On a besoin de Destroy. Le problème est que Destroy n'est pas directement exposé dans l'Inspector d'Unity.

    La solution la plus propre : un micro-script d'une seule méthode.

    2.3 Créer le script AutoDestroy

    Créez un script C# nommé AutoDestroy :

    using UnityEngine;

public class AutoDestroy : MonoBehaviour
{
    public void Detruire()
    {
        Destroy(gameObject);
    }
}

    Ce script ne fait qu'une chose — exposer Destroy à l'Inspector sous un nom lisible. C'est volontairement minimal.

    2.4 Brancher AutoDestroy dans l'Inspector

    Toujours en mode édition du prefab :

    1. Attachez le script AutoDestroy au prefab
    2. Sur le composant XR Simple Interactable → Select Entered
    3. Glissez le GameObject dans le champ objet
    4. Dans le menu déroulant, sélectionnez AutoDestroy → Detruire()
    5. Sauvegardez le prefab (Ctrl+S ou le bouton Save en haut)

    Ce qui se passe : Quand le contrôleur gauche pointe un cube peint et que la gâchette est pressée, le toolkit déclenche selectEntered sur ce cube précis, qui appelle Detruire(), qui appelle Destroy(gameObject). Le toolkit a géré la détection — on n'a pas eu besoin de raycast.

    Étape 3 : Simplifier PeintureVR

    La méthode OnGachetteGauche et sa référence actionGachetteGauche ne sont plus nécessaires.

    3.1 Modifier le script

    Retirez de PeintureVR.cs :

    • Le champ [SerializeField] private InputActionReference actionGachetteGauche
    • L'abonnement dans OnEnable : actionGachetteGauche.action.performed += OnGachetteGauche
    • Le désabonnement dans OnDisable
    • La méthode OnGachetteGauche complète

    Le script PeintureVR se retrouve avec une seule action input — actionGachetteDroite pour la peinture. C'est plus lisible et chaque responsabilité est clairement séparée.

    3.2 Nettoyer l'Inspector

    Sur le GameObject GestionnaireVR :

    • Le champ Action Gachette Gauche a disparu — c'est normal
    • Vérifiez que Action Gachette Droite pointe toujours vers XRI Right Interaction / Select

    Étape 4 : Tester

    Faites un nouveau build et vérifiez dans le casque :

    ✅ Le contrôleur droit pointe les cubes de couleur — la ligne s'arrête dessus ? ✅ Le contrôleur droit ne réagit plus aux cubes peints (pas de ligne courte sur les cubes peints) ? ✅ Le contrôleur gauche pointe un cube peint — la ligne s'arrête dessus ? ✅ Presser la gâchette gauche sur un cube peint le supprime ? ✅ Presser la gâchette gauche sur le mur ou dans le vide ne fait rien ? ✅ Le contrôleur gauche ne réagit pas aux cubes de couleur ?

    Récapitulatif des concepts ajoutés

    ✅ Interaction Layers : isoler quels interactors peuvent interagir avec quels interactables ✅ selectEntered : événement toolkit déclenché quand la gâchette confirme une sélection sur un interactable ciblé ✅ Événements dans l'Inspector : brancher une action directement sur un composant sans écrire de logique de détection ✅ Séparation des responsabilités : chaque contrôleur a un rôle distinct, chaque type d'objet a son layer

    Les deux approches côte à côte

                        [Exercice principal]
Gâchette gauche → Physics.Raycast → CompareTag("CubePeint") → Destroy()

                    [Cet exercice]
Gâchette gauche → Near-Far Interactor (layer Peint) → selectEntered → Detruire()

    Questions de réflexion

    1. Pourquoi selectEntered est-il sémantiquement plus juste que action.performed pour supprimer un cube ciblé ?
    2. Que se passerait-il si on oubliait de configurer les Interaction Layers et que les deux contrôleurs répondaient à tous les interactables ?
    3. Le script AutoDestroy est volontairement minimal — une seule méthode publique. Pourquoi est-ce une bonne pratique pour les scripts destinés à être utilisés dans l'Inspector ?
    4. Dans quel scénario préféreriez-vous revenir à l'approche Physics.Raycast plutôt qu'utiliser les événements du toolkit ?
